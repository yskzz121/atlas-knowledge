01

ドメインとは何か

ウェブサイトにアクセスするとき、ブラウザに atlas-financials.jp と入力しますよね。これが「ドメイン」です。

しかし、コンピュータ同士は実際には 185.199.108.153 のような数字の羅列（IPアドレス）で通信しています。ドメインは、人間が覚えやすいように付けた「名前」にすぎません。

メタファー

ドメインは「住所」、IPアドレスは「緯度経度」。あなたの家は「東京都港区赤坂1-1-1」で届くけれど、GPSでは「北緯35.6762°、東経139.7100°」。人間は住所を使い、機械は座標を使う。インターネットも同じ構造。

02

DNS — インターネットの電話帳

DNS（Domain Name System）は、ドメイン名をIPアドレスに変換するシステムです。誰かが atlas-financials.jp にアクセスすると、まずDNSに「このドメインのIPアドレスは？」と問い合わせが行きます。

あなた → 「atlas-financials.jp ってどこ？」 DNS → 「185.199.108.153 だよ」 あなた → そのIPアドレスにアクセス → サイトが表示される
メタファー

DNSは昔の「104番（電話番号案内）」。電話番号が分からないとき104に電話して「○○さんの番号を教えて」と聞いた。DNSはインターネット版の104番。

DNSの歴史

1983年以前
世界中のコンピュータの名前とIPアドレスが、たった1つのテキストファイル（HOSTS.TXT）に書かれていた。新しいコンピュータが追加されるたびに、世界中の人がそのファイルをダウンロードし直していた。
1983年
Paul Mockapetrisが「分散型電話帳」としてDNSを発明。1つのファイルではなく、世界中のサーバーが分担して名前解決を行う仕組みに。
現在
世界中に数千のDNSサーバーが存在。Google（8.8.8.8）やCloudflare（1.1.1.1）などの公開DNSが無料で利用可能。毎日数兆件の問い合わせを処理。
03

DNSレコードの種類 — 電話帳の「欄」

電話帳に「電話番号」「FAX番号」「住所」など複数の欄があるように、DNSにも複数の種類（レコードタイプ）があります。

レコード意味Atlas での設定例
A「このドメインの座標（IP）はここ」atlas-financials.jp → 185.199.108.153
CNAME「この名前は別の名前のエイリアス」www → atlasfinancials.github.io
NS「この電話帳の管理者は誰か」01.dnsv.jp（お名前.comのサーバー）
TXT「メモ欄（自由記述）」ドメイン認証コード
MX「メールの届け先」メールサーバーの指定
メタファー

Aレコードは「電話番号」。CNAMEは「こちらにお問い合わせください」という転送メモ。TXTは付箋紙に自由に書ける「メモ欄」。NSは「この電話帳はどの出版社が管理しているか」。

04

DNS浸透 — なぜ「待つ」必要があるのか

DNSレコードを変更しても、すぐには世界中で反映されません。世界中に何千ものDNSサーバーがあり、それぞれがレコードのコピーを持っています。全てのコピーが更新されるまでの時間をDNS浸透（propagation）と呼びます。

メタファー

引っ越して転居届を出しても、すぐには全ての郵便物が新しい住所に届かない。各郵便局が情報を更新するのに時間がかかる。DNSも同じで、通常は数分〜数時間、最大で24〜48時間。

今回の実体験では、お名前.comでTXTレコードを追加してから約3分で主要なDNSサーバー（Google、Cloudflare、Quad9、OpenDNS）への浸透が完了しました。ただし、特定のサービス（今回はGitHub）が独自にキャッシュを持っている場合、さらに時間がかかることがあります。

05

GitHub Pages — 無料のウェブサーバー

通常、ウェブサイトを公開するにはサーバー（コンピュータ）を借りて、そこにファイルを置く必要があります。月額数千円〜数万円かかります。

GitHub Pagesは、GitHubが提供する無料のホスティングサービスです。HTMLファイルをGitHubのリポジトリに入れると、自動的にインターネット上で公開してくれます。

メタファー

GitHub Pagesは「無料のレンタルギャラリー」。あなたの作品（HTMLファイル）をギャラリー（リポジトリ）に持ち込むと、看板を出して一般公開してくれる。デフォルトの看板（atlasfinancials.github.io）は無料。自分のブランド看板（atlas-financials.jp）を出す場合は身分証明が必要。

リポジトリにHTMLを置く ↓ GitHub Pages が自動でビルド ↓ atlasfinancials.github.io で公開 ↓ カスタムドメイン設定（CNAME） ↓ atlas-financials.jp で公開
06

ドメイン認証 — 「私が本物の持ち主です」

GitHub Organizationでカスタムドメインを使うには、ドメインの所有権を証明する必要があります。これがドメイン認証（Domain Verification）です。

GitHub: 「このコード（9f82ce...）をDNSに書いてください」 ↓ あなた: お名前.comでTXTレコードを追加 ↓ GitHub: 「DNSを確認... コードがある！ DNSを編集できるのはドメインの持ち主だけ。 あなたは本物の持ち主ですね ✓」
メタファー

ビルの管理会社（GitHub）が「あなたのブランドの看板を出すなら、身分証明をしてください」と言う。方法は「あなたしか貼れない場所（DNS）に、指定したコード（TXTレコード）を貼ってください」。貼れたということは、そのドメインの管理者 = 本物の持ち主。

この「特定の場所にコードを設置して所有権を証明する」パターンは、ウェブの世界で広く使われています。Google Search ConsoleやSSL証明書の発行でも同じ仕組みです。

07

SSL証明書 — 通信を暗号化する鍵

ブラウザのアドレスバーに鍵マークが表示されているサイトがありますよね。あれはSSL/TLS証明書が有効で、通信が暗号化されていることを意味します。

プロトコル安全性メタファー
http://（暗号化なし）危険ハガキ — 配達中に誰でも内容を読める
https://（暗号化あり）安全封書 — 封を開けないと中身が見えない

GitHub Pagesでは、ドメイン認証が完了するとSSL証明書を無料で自動発行してくれます（Let's Encryptという無料の認証機関を利用）。

08

今回の設定の全体像

Atlas Financialsのドメイン設定で実際に行ったことを時系列で整理します。

① atlas-financials.jp を取得（お名前.com） ↓ ② Aレコード設定 → GitHub Pages の IPアドレスへ向ける ↓ ③ CNAMEレコード設定 → www を atlasfinancials.github.io へ ↓ ④ GitHub Pages でCNAMEファイルを設置 ↓ ⑤ TXTレコード追加 → ドメイン認証コード ← 今ここ ↓ ⑥ GitHub でドメイン認証を完了 ↓ ⑦ SSL証明書（HTTPS）を有効化 ↓ ⑧ atlas-financials.jp で安全にアクセス可能に
